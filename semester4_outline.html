Day 1
What Can AI Do With Code? ‚Äî Demo, Discussion & Ground Rules
Intro
Teacher demonstrates AI code generation live: prompt an AI tool to write a simple JS function, then show the class the output. Immediately: "Does this work? How would we know?" Students discuss what they'd need to understand to verify it. Class establishes a shared AI Use Policy for the course: AI is a tool like a calculator ‚Äî powerful but not infallible. You are always the engineer responsible for the output. Students write their own "AI Use Commitment" in their notebooks.
Indiana Standards
IAS-CS 8.IC.1 IAS-CS 8.AP.3
IAS-CS 8.IC.1 (Impacts of Computing): Evaluate the beneficial and harmful effects of computing innovations. AI code generation is a direct and current example ‚Äî students analyze its benefits (speed, scaffold) and risks (errors, over-reliance, academic integrity).
Notebook Entry
Students write their "AI Use Commitment": 3 rules they personally commit to when using AI for coding. These are revisited and reflected on at the end of the semester.
Day 2
The Fact-Checking Framework ‚Äî 5 Steps to Verify AI Code
Intro
Teacher introduces the course's AI Code Fact-Checking Framework ‚Äî a 5-step process students will use every time they work with AI-generated code. Students annotate the framework in their notebooks and apply it to two provided AI-generated code snippets: one correct, one with a subtle bug. They must find the bug using the framework before running the code.
Indiana Standards
IAS-CS 8.AP.3 IAS-CS 8.IC.1
IAS-CS 8.AP.3: Identify and fix errors in algorithms and programs. The fact-checking framework is a systematic debugging methodology applied specifically to AI-generated code ‚Äî making error-detection a deliberate, structured skill.
The 5-Step Framework
AI Code Fact-Check Protocol
1
Read every line ‚Äî can you explain what each line does?
2
Trace the logic ‚Äî follow one input through to its output by hand
3
Test edge cases ‚Äî what happens with 0, negatives, or empty input?
4
Check names ‚Äî are variables and functions named meaningfully?
5
Run it ‚Äî compare actual output to your predicted output
Day 3
Prompt Engineering ‚Äî Getting Useful Code From AI
Skill
Students learn that AI output quality is directly tied to prompt quality. They practice writing engineering-specific prompts: specifying inputs/outputs, naming conventions, constraints, and expected behavior. They compare outputs from a vague prompt vs. a precise prompt for the same function. Class compiles a "Good Prompt Checklist" that lives in their notebooks. Key message: writing a good prompt requires you to already understand the problem ‚Äî the same skill as writing a good problem statement in the EDP.
Indiana Standards
IAS-CS 8.AP.1 IAS-CS 8.AP.2 IAS-CS 8.IC.1 CTE-TE 3.1
Prompt engineering is an extension of the EDP's "Define the Problem" step (CTE-TE 3.1) ‚Äî students must specify constraints, criteria, and expected outputs in their prompt just as they would in a design brief.

Day 4
AI-Assisted JS Project ‚Äî Build a Unit Converter
Project
Students use AI to help build a multi-function engineering unit converter (mm‚Üîin, kg‚Üîlbs, ¬∞C‚Üî¬∞F, PSI‚ÜîkPa). They must: write a detailed prompt, receive AI-generated code, apply the full 5-step fact-checking framework with written notes, identify and fix any issues, and then extend the converter with one original conversion not suggested by AI. The extension must be written entirely by the student ‚Äî no AI for that section. This two-part structure ensures AI is an accelerator, not a replacement.
Indiana Standards
IAS-CS 8.AP.1 IAS-CS 8.AP.2 IAS-CS 8.AP.3 IAS-CS 8.IC.1 IAS-Math 8.EE
Unit conversion is both a coding task (functions, DOM) and a math task (IAS-Math 8.EE ‚Äî applying proportional reasoning). The hybrid AI+manual approach ensures both standards are genuinely met.
Project Rubric (Formal)

Fact-check notes submitted
Original extension written by student
UI is clean and usable

Graded on completeness of annotation, accuracy of bug identification, and quality of correction. No AI tools permitted ‚Äî this assesses whether students can truly read and understand code.

Day 6
What is a Microcontroller? ‚Äî Arduino Hardware Orientation
Intro
Students unbox their Arduino Uno kits and identify all components: the board, USB cable, breadboard, jumper wires, LEDs, resistors, and sensors. Teacher explains the difference between a microcontroller (Arduino) and a computer ‚Äî focused single-purpose processing vs. general computing. Students label a diagram of the Uno in their notebooks: digital pins, analog pins, power pins (5V, 3.3V, GND), TX/RX, reset. Safety rules for electronics are established: never connect power without a resistor on an LED; always double-check wiring before powering.
Indiana Standards
CTE-TE 7.1 IAS-Sci 8.PS
CTE-TE 7.1: Apply principles of electricity and electronics to engineering systems. This standard anchors all Arduino work. IAS-Sci 8.PS connects through Ohm's Law ‚Äî students calculate required resistor values for their LED circuits.
Vocabulary
microcontroller digital pin analog pin GND 5V rail breadboard resistor Ohm's Law
Day 7
Blink ‚Äî Hello World of Hardware
Skill
Students wire their first circuit (LED + resistor to digital pin 13) and upload the classic Blink sketch. Teacher introduces the Arduino IDE and the structure of every Arduino program: setup() runs once, loop() runs forever. Students modify delay() values to change blink speed, then write their own version from scratch that blinks in a pattern (long-short-long = SOS). Notebook entry: wiring diagram drawn in the same format as technical drawings ‚Äî labeled components, pin numbers, and wire colors.
Indiana Standards
CTE-TE 7.1 IAS-CS 8.AP.1 IAS-Sci 8.PS
The loop() function is a direct physical manifestation of a programming loop (IAS-CS 8.AP.1) ‚Äî students see the connection immediately. The LED + resistor circuit applies Ohm's Law from IAS-Sci 8.PS.

Day 8
Digital Input ‚Äî Buttons, Pull-up Resistors & Conditionals
Skill
Students add a pushbutton to their circuit and use digitalRead() to detect presses. Introduction to pull-up resistors and why floating pins cause unreliable readings. Students write a sketch that turns an LED on when the button is held and off when released, then extends it to toggle on/off with each press (requiring a state variable). The state variable concept directly reinforces Semester 3's JavaScript variable work ‚Äî the same logic, new environment.
Indiana Standards
CTE-TE 7.1 IAS-CS 8.AP.1 IAS-CS 8.AP.2 IAS-Sci 8.PS
The toggle requires both a conditional (IAS-CS 8.AP.1) and a state variable (IAS-CS 8.AP.2) ‚Äî students apply both CS concepts in a physical context for the first time. Pull-up resistors connect to circuit analysis in IAS-Sci 8.PS.
Assessment (Informal)
Button circuit wired correctly
Toggle logic works
State variable used correctly
Day 9
Analog Input ‚Äî Reading Sensors with analogRead()
Skill
Students wire a potentiometer (or light sensor / temperature sensor depending on kit) to an analog pin and use analogRead() to get values 0‚Äì1023. They learn to map() this range to useful outputs (e.g., 0‚Äì255 for LED brightness using analogWrite() PWM). Students use Serial.println() to print sensor readings to the Serial Monitor ‚Äî their first use of serial communication for debugging. Notebook entry: graph of sensor values vs. physical position sketched by hand.
Indiana Standards
CTE-TE 7.1 IAS-CS 8.AP.1 IAS-CS 8.AP.2 IAS-Math 8.F
IAS-Math 8.F (Functions): The map() function is a direct application of a linear function ‚Äî input range mapped proportionally to output range. Students connect Arduino's map() to y = mx + b from math class.
Code Concept Preview

Day 10
AI-Assisted Arduino ‚Äî Prompt, Verify, Deploy
Skill
Students use AI to generate an Arduino sketch for a specified behavior (e.g., "LED blinks faster as light level increases"), then apply the 5-step fact-checking framework to the C++ output before uploading. Key challenge: AI sometimes generates syntactically correct code that doesn't match the intended behavior ‚Äî students must trace logic, not just syntax. This is the first time the AI framework is applied to a language other than JavaScript, reinforcing that the verification mindset is language-agnostic.
Indiana Standards
IAS-CS 8.AP.3 IAS-CS 8.IC.1 CTE-TE 7.1
Applying the fact-checking framework to C++/Arduino code fulfills IAS-CS 8.AP.3 in a new language context. Students discover that logic errors (wrong behavior) are harder to catch than syntax errors ‚Äî a key engineering insight.
Assessment (Informal)
Prompt is specific and complete
Fact-check notes written
Circuit behaves as intended

They must wire the circuit, write the sketch from scratch (no AI), upload it, and demonstrate correct behavior to the teacher. Graded on circuit correctness, code logic, variable use, and live demonstration. Wiring diagram must be drawn in notebook before powering the circuit.

Day 12
Project Launch ‚Äî Design an Enclosure for Your Arduino Circuit
Project
Students are challenged to design and 3D-print a functional enclosure for one of their Arduino circuits in SolidWorks. This is the first project that truly integrates all skills: EDP documentation, SolidWorks modeling, 3D printing, and electronics. They must measure their components (Arduino board, breadboard, LED placement), sketch the enclosure design by hand in their notebook using orthographic views, then model it in SolidWorks. Constraints: must have a lid, cable opening, and visible LED window.
Indiana Standards
CTE-TE 3.1 CTE-TE 3.2 CTE-TE 5.2 CTE-TE 6.1 CTE-TE 7.1
This project fulfills the most standards of any assignment so far ‚Äî it is a genuine multi-discipline engineering task. Five CTE-TE standards are simultaneously addressed: design process, modeling, drawing, digital tools, and electronics. A portfolio-worthy moment.

Model, Iterate & Print ‚Äî Enclosure Fabrication
Project
Students complete their SolidWorks model and submit for printing. While prints run, students refine their Arduino sketch ‚Äî adding a feature that wasn't in their original circuit (e.g., a startup LED pattern, a blink-speed sensor, a button-triggered message to Serial Monitor). This parallel workflow (CAD fabrication + code refinement) mirrors real product development where hardware and software teams work concurrently. AI may be used for the code feature with full fact-check documentation.

Day 15
Assembly Day ‚Äî Fit Circuit Into Enclosure
Project
Students receive printed enclosures and test fit their circuits. Many will discover fit issues ‚Äî components slightly too large, holes misaligned, LED window in the wrong position. This is intentional: students document fit issues in their notebooks, modify their SolidWorks model, and re-print if time allows. If not, they write a detailed "v2 improvement plan." The engineering lesson: real parts rarely fit perfectly the first time ‚Äî iteration is normal and expected.
Indiana Standards
CTE-TE 3.1 CTE-TE 3.2 CTE-TE 6.2
Fit issues and iteration directly fulfill CTE-TE 3.1 and 3.2 ‚Äî the design and prototype evaluation loop. Students experience why manufacturing tolerances (introduced in Semester 2) matter in the context of a real assembly.
